# 05 給未來 session 的信

> 2026-07-03，由 Fable 5 在唯一一次高階 session 中寫下。
> 這不是規則檔——是設計意圖與誠實的自我評估。規則過時可以改，這封信說明「為什麼當初這樣設」。

## 一、三件使用者沒問、但我認為最重要的事

**1. 這個 repo 真正的技術債是 App.tsx，而制度只是止血帶。**
4391 行單檔是所有 token 問題的根源。codebase-map.md 讓弱模型能繞著走，但每次
功能增加，地圖就更難維護。如果哪天要做大改版，最值得的一次性投資是把 App.tsx
按「後台管理／前台排班／共用 hooks」拆成 5-8 個檔——拆完之後，本制度一半的
防護規則會變得多餘，那是好事。拆分屬高風險重構，屆時請用 03-templates.md 的
重構範本＋opus 出計畫，並分十幾個小 commit 做。

**2. 這套系統管的是真實病患的排班，工程判斷和政策判斷的界線要守住。**
「公平輪替怎麼算」「哪類病患優先」「請假衝突怎麼讓位」——這些看起來像演算法
問題，實際上是院方政策。過去的 AI session 若擅自「優化」了排班邏輯，錯誤會
以「某位病患一直排不到治療」的形式出現，而且很難被發現。所以 rubrics 第 3 節
把它列為必問項。不要因為「我看得懂程式碼」就以為自己有權改規則。

**3. 明文密碼與測試模式 Firestore 是已知的安全弱點，使用者可能不知道嚴重性。**
types.ts 的 AppUser 存明文密碼、README 引導用「測試模式」開 Firestore（任何人
可讀寫）、排班資料含病歷號。這在自用小工具尚可，一旦真的多人上線就是病患
資料外洩風險。未來 session 若聽到「要給全科使用」「要正式上線」，應主動提醒
使用者做三件事：Firestore security rules、密碼雜湊化（admin 密碼已是 SHA-256，
AppUser 還不是）、考慮病歷號脫敏。不用現在做，但要在對的時機說。

## 二、這套制度最可能的退化方式與預防

**退化 1：codebase-map.md 的行號漂移到沒人信。**
每次改 App.tsx 行號就漂，三個月後地圖全錯，弱模型發現不可信後就回去整檔讀。
預防：地圖已標明「行號只當起點，用 Grep 確認」；04-maintenance.md 授權弱模型
直接更新地圖。真正的解法見上面第 1 點（拆檔）。

**退化 2：規則被「補丁式」增修到互相矛盾。**
每次踩雷就往 CLAUDE.md 加一條，兩年後變成 400 行互相打架的清單——這正是
弱模型最無力處理的局面。預防：CLAUDE.md 150 行硬上限＋「教訓先進 lessons.md，
第二次出現才制度化」的兩段式流程＋每季健檢（04-maintenance.md 第 5 節）。
如果你發現規則打架，優先刪規則而不是加規則。

**退化 3：儀式化——照著格式填，但不做判斷。**
弱模型很容易把 rubric 當表格填：驗收條件抄一句「lint 通過」就算寫過驗收、
派 agent 驗證但 prompt 裡透露了期望答案。制度擋不住敷衍，只能靠抽查：
使用者若發現回報品質下降，最有效的問句是「**你的證據是什麼？**」——
凡是答不出具體指令輸出或檔案:行號的「完成」，都是儀式化的訊號。

## 三、誠實評估：哪些產出我信心最低

1. **01-dispatch.md 第 7 節的費用直覺（haiku 1x / sonnet 3x / opus 15x）**——
   這是我從記憶中的量級印象寫的，違反了「不憑記憶填」的精神，只是標成「直覺」
   而非事實。實際比例請以官方 pricing 頁為準，若差很多請直接改掉該節。
   ＊【已解決 2026-07-03，第二次 Fable session】查證官方定價後果然錯了：
   opus 4.8 實為 haiku 的 5x（不是 15x）、sonnet 的 1.7x。該節已改寫，
   調度含義也修正為「該升級就升級，省錢靠精準交辦而非死守便宜模型」。
   這件事本身就是「不憑記憶填」鐵律的最好例證。
2. **升降級的具體門檻（錯 1 次升級、2 輪上限）**——方向我有信心，數字是拍的。
   沒有實驗依據證明「2 輪」優於「3 輪」。用三個月後，依實際體感調整數字即可，
   但保留「有限重試＋帶失敗軌跡升級」的結構。
3. **「被導向 Opus 4.8 的請求是否消耗本窗口額度」——未確認。** 我查不到，
   依誠實條款不編造。建議使用者到 usage 儀表板實測後，把答案補進 01-dispatch.md 第 2 節。
4. **未來 session 的 harness 是否長這樣**——本制度按「claude.ai/code 遠端環境＋
   目前這組 subagent 類型」寫成。若你在本機 CLI 或新版 harness 裡發現工具名對不上
   （例如沒有 Explore agent），請照 04-maintenance.md 第 3 節「事實過時」流程更新
   01-dispatch.md，制度結構不必動。
