# 02 判斷力外化：rubric 與 checklist

> 讀者：Sonnet / Haiku 等級 session。每條都有可勾選的判準＋一正例一反例。
> 用法：遇到對應情境時翻到該節，逐條核對，不憑感覺。

## 1. 何時升級模型（或建議使用者換模型）

符合**任一**條即升級（升級路徑見 `01-dispatch.md` 第 5 節）：
- [ ] 同一子任務已失敗 2 次，且兩次失敗原因不同（原因相同＝方向錯，見第 4 節，升級沒用）
- [ ] 任務要在「多個都說得通的方案」中選一個，且選錯的代價 > 重做一天（架構、資料 schema、刪除類）
- [ ] 需要讀懂 >3 個檔案的隱含互動才能下手（例：改輪替演算法同時牽動歸檔與 Firebase 同步）
- [ ] 使用者明說「這很重要，想清楚」

**正例**：改 `handleBatchAutoSchedule` 的公平性邏輯，牽動 rotationIndices、archive、
Firebase 三處狀態 → 派 opus 出方案，sonnet 實作。
**反例**：改按鈕文字失敗一次（打錯檔名）就升 opus → 浪費，那是執行錯，自己修正重試即可。

## 2. 何時算真的完成（Definition of Done）

**全部**打勾才能說「完成」：
- [ ] `npm run lint` 通過（貼出指令輸出的最後幾行為證）
- [ ] `npm run build` 通過
- [ ] 行為驗證：改了 UI/邏輯，至少描述「使用者操作 X 應看到 Y」並用 `/verify` 或請使用者確認；
      只過編譯不算行為正確
- [ ] 改了資料 schema → `src/migrations.ts` 已加對應 migration 並 bump 版本
- [ ] 已 commit 並 `git push -u origin <branch>` 成功、PR 存在
- [ ] 回報中如實寫「驗到哪一層」（編譯？行為？資料相容？）

**正例**：「已改完，lint/build 通過（輸出如下），行為面我用 dev server 點過新增病患流程，
schema 未動所以不需 migration，已 push 至 PR #12。」
**反例**：「已修改完成，應該可以用了。」——沒有任何證據，禁止這樣回報。

## 3. 何時停下來問使用者

符合**任一**條就停（用 AskUserQuestion，附你的建議選項）：
- [ ] 要刪除或覆寫「不是你這個 session 建立的」資料/檔案，且無備份
- [ ] 使用者的要求有 ≥2 種合理解讀，且做錯要重工 >30 分鐘
- [ ] 發現要動的東西和使用者描述的不符（他說「壞了」但你實測是好的）
- [ ] 涉及對外動作：寄信、發佈、開對外 PR 到別人的 repo、動 Firebase 正式資料
- [ ] 醫療排班規則本身的取捨（哪類病患優先）——這是院方政策，不是工程判斷

**反過來，這些不要問**：檔名怎麼取、commit message 怎麼寫、先做哪個子步驟、
用哪個現成工具——有慣例就照慣例，沒慣例就選了再在回報中說明。

**正例**：「『把輕度移除』可以指 (a) 移除類別選項 (b) 刪除所有輕度病患資料。(b) 不可逆，
我建議 (a)，請確認。」
**反例**：「請問 commit message 要用中文還是英文？」——看 git log 就知道（繁中），不要問。

## 4. 方向錯的訊號（該換路，而非重試/升級）

出現**任兩項**就停止重試，回頭重新拆問題：
- [ ] 連續兩次失敗的錯誤訊息幾乎相同
- [ ] 每修好一處就冒出新的一處壞掉（打地鼠）
- [ ] 修正的 diff 越來越大，但離驗收條件沒有更近
- [ ] 你開始想「把這個檢查關掉/註解掉就過了」——這是最強的紅燈
- [ ] 需要假設一個查不到證據的前提才能繼續（「應該是快取問題吧」）

換路的動作：把「原目標、已試路徑、失敗證據」寫下來，派 fresh-context agent
問「除了這條路還有什麼做法」，或直接回報使用者。
**正例**：三次調整 useEffect 依賴陣列都造成無限迴圈 → 停，重新檢視資料流設計，
發現該把同步邏輯搬出 effect。
**反例**：第四次微調依賴陣列「這次應該行」。

## 5. 品質底線怎麼驗（本 repo 專用）

- 任何 TS 改動：`npm run lint`（= tsc --noEmit）是底線，永遠先跑這個，最快。
- UI 改動：`npm run build` 之外，用 `npm run dev` + `/run` 或 `/verify` skill 實際打開頁面。
- 資料相容：改了 types.ts 或 localStorage key → 手動推演「舊資料載入新程式」路徑，
  必要時派 agent 寫一段 node 腳本模擬舊 JSON 過 migration。
- 文件/制度檔改動：派 fresh-context agent read-back（給它「應包含什麼」清單，不給結論）。
- 「看起來能跑」不是品質底線，「有證據能跑」才是。

**正例**：改 `PatientCategory` 後，同步查 `data.ts` 的 `generateInitialSchedule`、
`getCategoryLabel`、migration——Grep `住院複雜` 找出所有硬編碼點逐一核對。
**反例**：只改 types.ts 就宣告完成，舊備份匯入直接炸。
